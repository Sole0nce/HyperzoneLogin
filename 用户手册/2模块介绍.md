# 2. 模块介绍

本章的核心目标不是解释源码，而是回答一个更实际的问题：**你到底需要安装哪些模块？**

根据当前仓库结构，`HyperZoneLogin` 由核心模块与多个子模块组成。并不是所有模块都必须安装，正确做法是根据你的登录方案与迁移需求选择组合。

## 2.1 模块总览

当前仓库中可以确认的模块如下：

| 模块 | 作用 | 是否通常必需 |
|------|------|--------------|
| `velocity` | 主插件 / 核心运行时，负责 Profile、数据库、命令与模块注册 | 是 |
| `auth-yggd` | 提供 Mojang / 第三方 Yggdrasil 在线验证能力 | 视需求而定 |
| `auth-offline` | 提供本地 / 离线账号注册、登录、绑定、改密等能力 | 视需求而定 |
| `data-merge` | 从 AuthMe 或 MultiLogin 迁移数据 | 只在迁移时需要 |
| `profile-skin` | 在认证链路中提取、修复、缓存并应用皮肤属性 | 有皮肤需求时推荐 |
| `api` | 面向开发者的 API 项目，不是直接给服主放进 `plugins/` 的运行模块 | 否 |

## 2.2 先决定你的用户类型

### 方案 A：只做正版 / 第三方皮肤站验证

如果你的目标是：

- 只允许 Mojang 正版或第三方 Yggdrasil 玩家进入；
- 不提供离线本地账号注册；
- 希望围绕 Yggdrasil 档案做统一管理；

那么你至少应考虑：

- 核心主插件；
- `auth-yggd`；
- 如需完善皮肤表现，再加 `profile-skin`。

这种方案下，**不需要安装 `auth-offline`**。

### 方案 B：同时支持离线 + 正版 + 皮肤站验证

如果你的服务器既要接收离线玩家，又要兼容正版或第三方 Yggdrasil 玩家，那么至少需要：

- 核心主插件；
- `auth-yggd`；
- `auth-offline`。

如果你还希望在 Yggdrasil 登录后自动处理皮肤缓存与修复，再额外加入 `profile-skin`。

### 方案 C：从旧系统迁移过来

如果你之前已经在使用下列插件之一：

- `AuthMe`
- `MultiLogin`

并且希望把旧数据迁移进 HyperZoneLogin，那么在完成基础部署后，还需要安装：

- `data-merge`

如果你是第一次部署 HyperZoneLogin，没有任何旧数据需要导入，**就不需要安装这个模块**。

## 2.3 各模块说明

### `velocity`：核心模块

根据 `velocity/README.md`，这是 HyperZoneLogin 的主插件，负责以下核心能力：

- 管理玩家档案（Profile）的注册、查找与绑定；
- 提供主运行时 API；
- 提供数据库管理与表定义；
- 暴露模块注册入口，让子模块接入统一框架；
- 提供基础管理命令，例如重载与查看玩家 / Profile 信息。

你可以把它理解为整套系统的“底座”。没有它，其他模块都无从挂载。

### `auth-yggd`：Yggdrasil 在线验证模块

根据 `auth-yggd/README.md`，该模块主要负责：

- 从 `entry/` 目录读取多个 Yggdrasil 配置；
- 向 Mojang 或第三方 Yggdrasil 服务发起验证；
- 并发处理多个 Entry 的验证请求；
- 在验证成功后，把 Entry 记录与本地 Profile 关联。

适用场景：

- 正版验证；
- 第三方皮肤站验证；
- 一个服务器同时接入多个 Yggdrasil 服务。

如果你的服务器完全不打算做正版 / 皮肤站验证，这个模块就没有必要安装。

### `auth-offline`：离线 / 本地账号模块

根据 `auth-offline/README.md`，该模块提供的是“服务器内部账号体系”，包括：

- `/login`
- `/register`
- `/bind`
- `/changepassword`
- `/logout`
- `/unregister`

它适用于以下情况：

- 你需要允许离线玩家进入；
- 你希望玩家使用服务器本地账号密码完成登录；
- 你需要对离线账号做注册、改密、注销等操作。

如果你的服完全只做在线验证，不提供离线入口，那么这个模块不是必需的。

### `data-merge`：迁移模块

根据 `data-merge/README.md`，这个模块专门用于把旧数据迁移到 HyperZoneLogin 的数据模型中，目前明确支持：

- MultiLogin / MinecraftLink 旧数据迁移；
- AuthMe 数据迁移。

模块会提供 `/hzl-merge` 命令，并读取 `merge/merge-ml.conf` 与 `merge/merge-am.conf` 中的迁移配置。

这个模块的重点不是“日常运行”，而是“历史数据接管”。因此：

- **有旧数据**：装；
- **没有旧数据**：不装。

### `profile-skin`：皮肤修复模块

根据 `profile-skin/README.md`，这个模块会在 `auth-yggd` 成功认证后：

- 提取上游 `GameProfile` 中的 `textures`；
- 优先缓存已签名皮肤数据；
- 对未签名但可解析的皮肤源尝试修复；
- 将结果写入缓存表；
- 在最终转发给后端时把皮肤属性重新应用回去。

适用场景：

- 你使用 `auth-yggd`；
- 你希望皮肤属性在代理转发链路里更稳定；
- 你希望尽量兼容上游皮肤信息不完整的情况。

如果你当前只想先跑通基本登录链路，可以暂时不装；如果你很在意皮肤表现与档案完整性，建议一并部署。

## 2.4 推荐安装组合

### 最小正版 / 皮肤站方案

- 主插件
- `auth-yggd`

可选：

- `profile-skin`

### 混合登录方案（离线 + 正版 / 皮肤站）

- 主插件
- `auth-yggd`
- `auth-offline`

可选：

- `profile-skin`
- `data-merge`（仅迁移时）

### 旧服迁移方案

基础运行模块：

- 主插件
- 按你的目标登录形态安装 `auth-yggd` / `auth-offline`

迁移附加模块：

- `data-merge`

## 2.5 一个简单的判断原则

如果你还拿不准自己该装什么，可以按下面这个顺序判断：

1. **先装核心主插件**；
2. 需要在线验证，就装 `auth-yggd`；
3. 需要离线本地账号，就装 `auth-offline`；
4. 需要迁移历史数据，才装 `data-merge`；
5. 需要皮肤修复与缓存，再装 `profile-skin`。

这样选出来的组合，通常就是最接近实际需求的最小集。
