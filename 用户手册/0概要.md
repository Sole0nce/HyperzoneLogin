# 0. 概要

## 项目定位

`HyperZoneLogin` 是一个面向 Minecraft 代理服场景的多登录方式框架，用于统一管理不同来源的玩家身份验证、档案（Profile）与相关扩展能力。按照仓库根目录 `README.md` 的描述，它的目标是“实现多种登入方式的框架”，并已经覆盖以下几类核心能力：

- 取代传统离线登录插件的本地账号验证；
- 取代传统多 Yggdrasil 档案管理方案；
- 为档案链路提供皮肤修复能力；
- 提供 AuthMe、MultiLogin 数据导入能力；
- 通过独立 `api` 与主插件运行时接口支持进一步扩展。

## 诞生背景

这类框架的需求，通常来自两个方向的共同推动：

1. **服主侧需求**：希望在 Velocity 代理层统一处理登录逻辑，而不是在后端服拼接多套插件；
2. **玩家侧需求**：希望在正版、第三方皮肤站、离线账号等不同接入方式之间获得更一致的登录体验。

从现有仓库规划可以看出，项目关注的不是“只支持一种登录方式”，而是“把多种验证入口整合到同一套档案模型中”，让登录、迁移、皮肤处理与后续扩展都能围绕同一个核心运行。

## 设计特点

### 1. 面向代理层的数据处理思路

根据 `ai生成词` 中的原始要求，本项目强调采用类似协议层数据处理的方式，而不是依赖传统的高耦合反射替换思路。对服主而言，这带来的实际意义是：

- 结构更清晰，模块边界更明确；
- 后续扩展更容易围绕统一事件与 API 进行；
- 不同登录方式可以在同一个代理层框架内组合，而不是互相叠补。

这里更值得关注的，不是“宣传某一种技术路线绝对更强”，而是它确实围绕代理服登录场景做了**统一建模**：先识别玩家身份，再将其映射到 `Profile`，最后把档案结果传递给后端服务器。

### 2. 模块化设计

项目采用了清晰的多模块结构。根据 `settings.gradle.kts`，当前模块包括：

- `velocity`：主插件/核心模块；
- `api`：对外 API；
- `auth-yggd`：Yggdrasil 在线验证模块；
- `auth-offline`：离线/本地账号模块；
- `data-merge`：迁移模块；
- `profile-skin`：皮肤修复模块。

这种设计的最大价值，是让服主可以按需求安装：

- 只做正版或皮肤站验证，就安装核心 + `auth-yggd`；
- 需要同时支持离线登录，就再加入 `auth-offline`；
- 正在从旧系统迁移，才需要 `data-merge`；
- 需要补足皮肤链路，再使用 `profile-skin`。

这比“一个大而全的单体插件”更适合实际运维，因为你只需要启用自己真的会用到的部分。

### 3. 可扩展 API 与原型实现

从 `velocity/README.md` 可以确认，主插件会向其他模块暴露运行时 API，并支持子模块通过 `registerModule(...)` 进行注册。也就是说，本项目不仅提供现成功能，还提供了一套可供扩展的原型：

- 主插件负责数据库管理、命令/事件集成点、Profile 表定义等基础能力；
- 子模块围绕这些能力扩展各自的认证与处理逻辑；
- 如果你准备实现新的验证方式，也可以参考现有模块源码进行适配。

这使得 `HyperZoneLogin` 不只是一个插件集合，更像是一套围绕“玩家身份与档案”展开的登录框架。

## 当前可确认的兼容范围

基于根目录 `README.md`，当前能明确确认的信息包括：

- **可导入数据来源**：AuthMe、MultiLogin；
- **Minecraft 版本**：`1.19.1+` 暂无问题；
- **低版本提示**：低于 `1.19.1` 时，只能使用域名作为识别方式；
- **启动器支持**：支持市面常见启动器。

需要注意的是，这里是仓库当前公开写明的范围，不等于完整兼容矩阵。更细的服务端实现、代理组合、皮肤站适配情况，仍建议结合测试环境自行验证。

## 适合哪些使用场景

这套框架特别适合以下几类服务器：

- 想在 Velocity 代理层统一处理登录验证的服务器；
- 同时面对正版、第三方 Yggdrasil 与离线玩家的混合型服务器；
- 正在从 AuthMe 或 MultiLogin 迁移到新架构的服务器；
- 希望后续继续接入自定义验证逻辑、Web 管理或更多扩展功能的服务器。
