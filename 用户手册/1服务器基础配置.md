# 1. 服务器基础配置

本章只讲一个前提：**先把 Velocity 代理与后端服的基础转发链路配置正确，再谈 HyperZoneLogin。**

如果你对 Velocity、Paper、`forwarding.secret`、后端服 `online-mode` 这些概念还不熟悉，建议先补齐代理服基础知识。HyperZoneLogin 是建立在这些前置条件之上的登录框架，而不是替你跳过这些基础配置的“免配置插件”。

## 1.1 为什么必须先完成基础配置

`HyperZoneLogin` 工作在代理层，需要可靠地识别玩家、控制认证阶段的转发行为，并把最终档案结果送到后端服务器。这要求你的代理链路至少满足以下条件：

- Velocity 与后端服之间的转发方式正确；
- 后端服只接受来自代理的连接；
- 代理与后端使用一致的密钥；
- 后端不会自行以在线模式再次接管登录流程。

只要这几个前提有一个没有满足，后续出现“UUID 不对”“玩家被踢”“认证状态丢失”“后端直接进服”等问题都很常见。

## 1.2 推荐使用 modern forwarding

`ai生成词` 中已经给出了 Velocity 官方文档的核心结论：

- `modern forwarding` 是 Velocity 原生的转发格式；
- 它会在登录阶段以二进制格式转发玩家信息，并使用密钥校验；
- 它适用于 `Minecraft 1.13+`；
- 低于 `1.13` 或使用 `ProtocolSupport` 的环境，不适合这一模式。

对于 HyperZoneLogin 这类代理层登录框架，优先建议按现代转发方案配置。这样更符合当前主流 Velocity 代理服的部署方式，也更容易保证玩家身份数据在代理与后端之间保持一致。

## 1.3 Velocity 端配置

打开 Velocity 根目录中的 `velocity.toml`，确认如下配置：

```toml
player-info-forwarding-mode = "modern"
```

然后记录 Velocity 根目录下 `forwarding.secret` 文件中的内容。这个密钥稍后要原样填写到后端服配置中。

### 检查点

完成这一部分后，请确认：

- `player-info-forwarding-mode` 已经是 `modern`；
- `forwarding.secret` 文件存在；
- 你知道自己现在使用的是哪一个代理实例；
- 你的后端服确实是通过这个代理进入，而不是玩家能直连后端端口。

## 1.4 后端服配置

以下示例来自 `ai生成词` 中给出的配置流程，适用于常见 Paper 后端部署思路。

### `config/paper-global.yml`

确保 Velocity 转发已启用，并填写与代理一致的密钥：

```yaml
proxies:
  velocity:
    enabled: true
    online-mode: false
    secret: GByNEDvhjFhS
```

其中 `secret` 必须替换为你自己 `forwarding.secret` 文件中的真实内容。

### `server.properties`

确保后端服不会在代理后面再次以在线模式独立校验玩家：

```properties
online-mode=false
enforce-secure-profile=false
```

## 1.5 配置完成后的预期状态

当代理与后端服配置正确后，应该达到如下状态：

1. 玩家连接的是 Velocity，而不是直接连接后端；
2. 后端服能接收到来自 Velocity 的转发信息；
3. Velocity 与后端服通过同一个 `secret` 建立可信关系；
4. 后端服不会绕过代理重新执行自己的在线验证流程。

这时，HyperZoneLogin 才有稳定的运行基础。

## 1.6 与 HyperZoneLogin 直接相关的后端等待区配置

根据 `velocity/README.md` 与 `config/backend-server.conf`，如果你**没有安装 limboapi**，主插件可以把一个真实后端服务器作为认证等待区使用。相关配置包括：

```hocon
fallback-auth-server=lobby
post-auth-default-server=play
remember-requested-server-during-auth=true
```

它们的含义是：

- `fallback-auth-server`：未安装 `limboapi` 时，玩家在认证前会被送到哪个真实后端服等待；
- `post-auth-default-server`：认证完成后优先进入的服务器；
- `remember-requested-server-during-auth`：如果玩家认证期间尝试切换目标服，是否在认证后自动接续。

注意，这一部分**不是** Velocity 自身的基础配置，而是 HyperZoneLogin 在“无 limboapi 场景”下的补充行为配置。

## 1.7 常见误区

### 误区 1：后端服能进就说明代理配置没问题

不对。后端能进，只能说明服务端在监听；不能说明代理转发、密钥、在线模式关系完全正确。

### 误区 2：忘记同步 `secret` 也没关系

不对。`forwarding.secret` 与后端 `paper-global.yml` 中的 `secret` 必须一致，否则后端无法正确接受 Velocity 转发信息。

### 误区 3：后端继续开 `online-mode=true` 更安全

在代理统一接管验证的部署模型里，这通常会造成流程冲突。对当前架构，应让代理层统一处理，后端按代理转发结果工作。

### 误区 4：不懂代理基础也可以先把插件装上再说

不建议这样做。登录框架本身依赖代理基础知识；如果连代理转发模式、后端在线模式、连接路径都没有梳理清楚，排障成本会非常高。

## 1.8 建议的自检清单

在进入下一章之前，请逐项确认：

- [ ] 已完成 Velocity 与后端服的基础联通；
- [ ] `velocity.toml` 使用 `modern` 转发；
- [ ] `forwarding.secret` 已同步到后端；
- [ ] 后端 `online-mode=false`；
- [ ] 玩家无法绕过代理直连后端；
- [ ] 你知道认证等待区准备放在哪个后端服务器上。

这些内容如果还没有确认完，请先停在这一章，不要继续安装 HyperZoneLogin。
